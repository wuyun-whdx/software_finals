# 好友匹配交友功能 — 需求文档

> 状态：✅ 需求已确认，待进入开发  
> 归属：匹配模块（Match）的子功能扩展  
> 目标：在现有双人匹配基础上，构建好友关系管理、开放人格匹配推荐、好友即时聊天三大子系统

---

## 一、功能概述

在现有"双人匹配"（通过手机号/邀请码生成适配报告）的基础上，新增下面三个互相协作的子功能：

| 子系统 | 一句话描述 |
|--------|-----------|
| **好友关系管理** | 用户通过邀请码或好友申请建立好友关系，支持同意/拒绝/拉黑/删除 |
| **开放匹配推荐** | 用户授权将大五人格最终分数放入匹配池，系统推荐 ≥80% 兼容度的其他用户 |
| **好友聊天** | 在好友关系基础上提供一对一即时聊天能力，支持消息历史与未读提醒 |

### 核心理念

> 先有"了解"（适配报告），再有"关系"（加好友），再能"互动"（聊天）。

现有 Match 模块解决的是 **"我们合不合适"**（单次计算），新增功能解决的是 **"合适之后怎么办"**（建立关系、持续互动）。

## 二、核心数据规则（重要）

### 2.1 开放匹配的双向授权原则

| 规则 | 说明 |
|------|------|
| **不授权 = 不可见** | 用户未开启开放匹配时，该用户**不会出现在任何人的推荐列表**中 |
| **不授权 = 看不到** | 用户未开启开放匹配时，该用户**也看不到推荐列表**（无法查看自己与他人的兼容度） |
| **双向授权才匹配** | 只有双方都开启了开放匹配，才会互相出现在对方的推荐列表中 |
| **授权即入池** | 开启匹配后，用户的**10维大五人格最终分数**（仅最终测评结果，不含具体答题内容）进入匹配池 |
| **关闭即清退** | 关闭匹配后，该用户在匹配池中的数据**立即移除**，不保留任何历史痕迹 |
| **重新开启 = 重新入池** | 用户可随时重新开启，系统以当时的最终测评结果重新入池 |

### 2.2 数据隐私边界

```
完整答题内容         → 仅用户自己可见（现有 TestResult 表，不变）
10维最终分数         → 用户自己 + 开放匹配池（开启时）
适配报告            → 双人匹配双方可见（现有逻辑，不变）
好友关系            → 双方可见
聊天内容            → 对话双方可见
```

**隐私底线**：任何情况下，其他用户都无法看到你的具体答题内容、原始选项或中间计算过程。开放匹配只暴露最终的10维分数。

## 三、用户故事

### 3.1 好友关系管理

#### U1 — 通过邀请码添加好友

> **前置条件**：双方均已完成基础性格测试（PERSONALITY）。用户A已登录。

1. 用户A在好友页面点击"生成好友邀请码"
2. 系统生成一个唯一邀请码（格式：`FR` + 8位大写字母数字，如 `FR7B3A9E1`），有效期90天
3. 用户A将邀请码通过任意渠道分享给用户B
4. 用户B在好友页面输入该邀请码，点击"添加好友"
5. 系统验证邀请码有效 → **直接建立双向好友关系**（无需审批）
6. 邀请码标记为"已使用"，双方好友列表各自出现对方

**邀请码规则**：
- 同一时间每个用户只能有1个活跃的好友邀请码
- 生成新邀请码时，旧邀请码自动失效
- 邀请码与匹配邀请码（`ME` 前缀）**互不通用**
- 好友邀请码只建立关系，不生成适配报告；如需适配报告走现有 `/api/matches` 流程

#### U2 — 发送好友申请

> **前置条件**：申请人已登录，已知对方手机号或在推荐列表中看到对方。

1. 用户A通过手机号搜索用户B
2. 系统返回用户B的基本信息（昵称、头像）
3. 用户A点击"添加好友"，可选填写验证消息（最多50字）
4. 系统发送好友申请，状态为"待处理"
5. 用户B收到好友申请通知（红点/角标）

**约束**：
- 已是好友 → 不可重复申请
- 已被对方拉黑 → 不可发送申请
- 已有待处理申请 → 不可重复发送
- 不能给自己发申请

#### U3 — 处理好友申请

> **前置条件**：接收方已登录，存在待处理的好友申请。

1. 用户B在好友申请的列表中看到用户A的申请（含昵称、头像、验证消息、申请时间）
2. 用户B可以选择：
   - **同意** → 双方建立好友关系，申请状态变为"已同意"
   - **拒绝** → 申请状态变为"已拒绝"，不建立关系
   - **拉黑** → 申请状态变为"已拉黑"，用户A被加入黑名单，用户A无法再向用户B发送申请
3. 处理完毕后，该申请从待处理列表消失

#### U4 — 删除好友

1. 用户在好友列表中选择好友
2. 点击"删除好友"
3. 系统弹出确认对话框："删除后双方将不再互为好友，聊天记录仍保留。确定删除？"
4. 确认 → 好友关系标记为"已删除"，双方各自从好友列表移除对方
5. 删除后仍可重新发送好友申请

#### U5 — 拉黑管理

1. 用户可以在好友申请中直接拉黑，也可以在好友列表中拉黑已有好友
2. 拉黑后：
   - 好友关系自动断开
   - 被拉黑者无法向拉黑者发送好友申请（提示"无法添加该用户为好友"）
3. 用户可以查看自己的黑名单
4. 可以手动取消拉黑（解除后对方可以重新发送申请）

### 3.2 开放匹配推荐

#### U6 — 开启/关闭开放匹配

> **前置条件**：用户已完成基础性格测试（PERSONALITY），拥有有效的10维人格分数。

1. 用户在好友页面的"开放匹配"卡片中看到自己的匹配状态
2. 初始状态：**关闭**
3. 点击"开启匹配" → 弹出授权确认对话框：
   > "开启后，您的10维人格测评分数将进入公共匹配池，其他开启匹配的用户可以看到您的昵称、头像和人格分数。您的具体答题内容不会被展示。确定开启？"
4. 确认 → 用户人格数据入池，状态变为"已开启"
5. 点击"关闭匹配" → 弹出确认对话框：
   > "关闭后，您的数据将从匹配池中移除，历史匹配记录不会保留。确定关闭？"
6. 确认 → 数据从匹配池移除，状态变为"已关闭"

**规则**：
- 开启/关闭无冷却时间，可随时切换
- 关闭后立即生效，推荐列表实时更新

#### U7 — 查看匹配推荐列表

> **前置条件**：用户已开启开放匹配。

1. 用户进入"推荐匹配"标签页
2. 系统展示推荐用户列表，每人显示：
   - 昵称、头像
   - **兼容度百分比**（大号数字，≥80% 为绿色高亮）
   - Top3 高契合维度（如"开放性、饮食探索、社交能量契合度很高"）
   - "发送好友申请"按钮
3. 列表按兼容度从高到低排序
4. 已经是好友的用户不出现在推荐列表中
5. 已被拉黑的用户不出现在推荐列表中
6. 用户自己不在推荐列表中

**匹配计算逻辑**：
```
Step 1: 获取当前用户的 10 维人格分数
Step 2: 查询所有 enabled=true 的 OpenMatchProfile（排除自己）
Step 3: 排除已是好友的用户
Step 4: 排除已被当前用户拉黑的用户
Step 5: 对每个候选用户，调用 MatchEngine.compatibilityScore() 计算兼容度
Step 6: 过滤 score < 80% 的用户
Step 7: 按 score 降序排列，取 Top 20
```

#### U8 — 从推荐列表快速添加好友

1. 用户在推荐列表中点击某用户的"发送好友申请"
2. 弹出窗口，可输入验证消息（最多50字，预填"你好！系统推荐我们人格契合度很高，交个朋友吧～"）
3. 发送 → 创建好友申请，走 U2/U3 流程

### 3.3 好友聊天

#### U9 — 查看聊天列表

> **前置条件**：用户已登录，至少有一个好友。

1. 用户点击底部导航栏或侧边栏的"聊天"入口
2. 展示聊天列表，每个对话项显示：
   - 好友头像、昵称
   - 最后一条消息的摘要（最多20字）
   - 最后消息时间
   - **未读消息角标**（红色数字）
3. 列表按最后消息时间倒序排列（最新消息在上）
4. 无好友时显示空状态："还没有好友，去添加好友开始聊天吧～"

#### U10 — 一对一聊天

> **前置条件**：用户已登录，与聊天对象互为好友。

1. 点击聊天列表中的好友 → 进入聊天详情页
2. 页面组成：
   - 顶部栏：好友昵称 + 返回按钮
   - 消息区域：对话气泡（自己发右侧蓝色，好友发左侧灰色）
   - 底部输入区：文本输入框 + 发送按钮
3. 消息支持：
   - **文本消息**（1-500字）
   - 自动换行、空格保留
4. 发送后消息立即显示在对话区
5. 消息状态：已发送（单勾）/ 对方已读（双勾）
6. 页面打开时自动加载最近50条历史消息
7. 向上滚动加载更早的消息（分页加载，每页30条）
8. 新消息通过**短轮询**（每3秒拉取一次）获取

#### U11 — 未读消息提醒

1. 聊天列表页的每个对话显示未读计数
2. 底部导航栏"聊天"入口显示总未读数角标
3. 进入聊天详情页后，自动将所有未读消息标记为已读
4. 在聊天详情页期间收到新消息，实时标记为已读

## 五、API 接口设计

所有接口遵循现有 `ApiResponse<T>` 统一响应格式。除标注外均需 `Authorization: Bearer <JWT>`。

### 5.1 好友关系接口

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/friends/by-invite` | 通过邀请码添加好友（直接建立关系） |
| `POST` | `/api/friends/invite` | 生成好友邀请码 |
| `POST` | `/api/friends/request` | 发送好友申请 |
| `GET` | `/api/friends` | 好友列表（含最后消息摘要） |
| `GET` | `/api/friends/requests` | 待处理申请（收到的+发出的） |
| `PUT` | `/api/friends/requests/{id}/accept` | 同意申请 |
| `PUT` | `/api/friends/requests/{id}/reject` | 拒绝申请 |
| `PUT` | `/api/friends/requests/{id}/block` | 拉黑申请人 |
| `DELETE` | `/api/friends/requests/{id}` | 取消发出的申请 |
| `DELETE` | `/api/friends/{friendId}` | 删除好友 |
| `GET` | `/api/friends/blocked` | 拉黑列表 |
| `PUT` | `/api/friends/{friendId}/unblock` | 取消拉黑 |
| `GET` | `/api/friends/search?phone=` | 通过手机号搜索用户 |

### 5.2 开放匹配接口

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/match/open/toggle` | 开启/关闭开放匹配 |
| `GET` | `/api/match/open/status` | 查询当前用户的匹配状态 |
| `GET` | `/api/match/open/recommendations` | 获取推荐用户列表（≥80%兼容度） |

### 5.3 聊天接口

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/chat/conversations` | 聊天列表（所有好友+最后消息+未读数） |
| `GET` | `/api/chat/messages/{friendId}?page=&size=` | 获取与某好友的聊天记录（分页） |
| `POST` | `/api/chat/messages/{friendId}` | 发送消息给某好友 |
| `PUT` | `/api/chat/messages/{friendId}/read` | 标记与该好友的所有未读消息为已读 |
| `GET` | `/api/chat/unread-count` | 获取总未读消息数 |

## 六、数据模型设计

### 6.1 新增实体

#### Friendship（好友关系）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | Long (PK, 自增) | 主键 |
| `user` | ManyToOne → UserAccount | 关系持有者 |
| `friend` | ManyToOne → UserAccount | 好友 |
| `status` | Enum: ACTIVE / BLOCKED / DELETED | 关系状态 |
| `createdAt` | Instant | 建立时间 |
| `updatedAt` | Instant | 最后更新时间 |

> 双向存储：A加B → 创建两条记录 `(user=A, friend=B)` 和 `(user=B, friend=A)`。

#### FriendRequest（好友申请）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | Long (PK, 自增) | 主键 |
| `fromUser` | ManyToOne → UserAccount | 申请人 |
| `toUser` | ManyToOne → UserAccount | 接收人 |
| `status` | Enum: PENDING / ACCEPTED / REJECTED / BLOCKED | 申请状态 |
| `message` | varchar(50) | 验证消息 |
| `createdAt` | Instant | 申请时间 |
| `updatedAt` | Instant | 最后更新时间 |

#### OpenMatchProfile（开放匹配档案）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | Long (PK, 自增) | 主键 |
| `user` | OneToOne → UserAccount | 用户 |
| `enabled` | boolean | 是否开启匹配 |
| `personalityJson` | @Lob TEXT | 10维人格分数 JSON |
| `updatedAt` | Instant | 最后更新时间 |

> `enabled=false` 时该记录保留但查询时排除（等价于从匹配池移除）。

#### ChatMessage（聊天消息）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | Long (PK, 自增) | 主键 |
| `sender` | ManyToOne → UserAccount | 发送者 |
| `receiver` | ManyToOne → UserAccount | 接收者 |
| `content` | TEXT | 消息内容 |
| `read` | boolean (默认 false) | 是否已读 |
| `createdAt` | Instant | 发送时间 |

索引：
- `idx_chat_conversation` (sender_id, receiver_id, created_at DESC)
- `idx_chat_receiver_unread` (receiver_id, read, created_at)

## 八、非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | 聊天消息轮询间隔 ≤ 3秒；推荐列表计算 ≤ 2秒（候选用户 ≤ 1000人时） |
| 安全 | 所有接口需 JWT 认证；开放匹配仅暴露10维分数，不可追溯原始答题 |
| 可用性 | 操作需二次确认（删除好友、关闭匹配、拉黑等） |
| 数据 | 关闭匹配后数据立即不可查询；删除好友后聊天记录保留 |
| 兼容 | 移动端优先，聊天界面适配 375px 宽度 |

## 九、测试要点

| 测试场景 | 验证点 |
|----------|--------|
| 邀请码加好友 | 生成 → 分享 → 对方输入 → 直接成为好友 |
| 好友申请流程 | 发送 → 收到通知 → 同意/拒绝/拉黑 |
| 重复申请 | 已是好友/已有待处理 → 拒绝发送 |
| 拉黑机制 | 拉黑 → 对方无法申请 → 取消拉黑 → 可重新申请 |
| 删除好友 | 删除 → 双方列表移除 → 聊天记录保留 |
| 开放匹配开关 | 开启 → 入池 → 关闭 → 出池 → 重新开启 → 重新入池 |
| 双向授权 | 只有双方都开启才互相出现在推荐中 |
| 推荐列表 | 只展示 ≥80% 用户 → 排除好友 → 排除已拉黑 → 按兼容度排序 |
| 聊天发送/接收 | 发送 → 对方收到 → 未读计数 → 进入对话标记已读 |
| 聊天分页 | 加载历史消息 → 上滑加载更多 |
